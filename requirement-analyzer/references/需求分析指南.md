# 需求分析指南

> 本文档提供需求分析的详细指导原则和方法论（精简版 - 聚焦业务分析）
> 作者：刘丰荣

---

## 1. 需求分析概述

### 1.1 什么是需求分析

需求分析是将用户原始需求转化为结构化需求文档的过程，聚焦于业务层面，不涉及技术实现。

### 1.2 需求分析的目标

1. **明确功能类型**：确定是纯定制还是基于产品个性化
2. **提取业务故事线**：描述用户操作流程
3. **梳理功能清单**：按优先级列出功能
4. **识别不确定项**：发现需求中的模糊点

---

## 2. 功能类型识别

### 2.1 纯定制

**定义**：从零开始开发的全新功能，不基于任何现有产品模块。

**特征**：
- 新建功能，无历史包袱
- 数据库表需要全新创建
- 代码命名无需特殊前缀

### 2.2 基于产品个性化

**定义**：基于Epoint现有产品模块进行定制开发。

**特征**：
- 继承产品基础功能
- 在产品表基础上扩展字段
- 代码需要使用Gxh前缀区分

---

## 3. 图片内容分析

### 3.1 什么是图片内容分析

当需求文档包含图片时，需要结合图片内容进行分析，识别页面布局、字段类型、交互元素等。

### 3.2 图片类型与分析要点

| 图片类型 | 分析要点 |
|----------|----------|
| 页面截图 | 页面布局、功能区域、字段位置 |
| 流程图 | 业务流程步骤、状态流转逻辑 |
| 原型图 | 字段类型、表格列、弹窗布局 |
| UI标注 | 必填/可选、输入格式、校验规则 |

### 3.3 分析原则

1. 图片分析与文字描述相结合
2. 如图片与文字描述有冲突，以图片为准
3. 无法从图片识别的内容，列入不确定项

---

## 4. 业务故事线

### 4.1 什么是业务故事线

业务故事线是描述用户完成一个业务目标所经历的操作序列。

### 3.2 业务故事线要素

| 要素 | 说明 |
|------|------|
| 角色 | 谁在操作（业务员、管理员等） |
| 触发 | 什么情况下操作（点击按钮、收到通知等） |
| 步骤 | 具体操作步骤（1、2、3...） |
| 结果 | 操作后的结果（系统反馈、数据变化） |

### 3.3 业务故事线示例

```
【业务故事线：场所信息新增】

角色：窗口业务员
触发：需要新增一个场所窗口信息
步骤：
  1. 点击"新增"按钮
  2. 填写场所基本信息（名称、地址、类型）
  3. 填写场所详细信息（负责人、联系电话）
  4. 上传场所资质图片
  5. 点击"保存"按钮
结果：系统提示"保存成功"，刷新列表显示新记录
```

### 3.4 提取原则

1. **每个核心功能至少1条故事线**
2. **覆盖主要业务流程**
3. **描述用户视角的操作**
4. **包含系统反馈**

## 5. 功能清单

### 5.1 功能清单格式【按页面拆分描述】

**核心原则**：每个功能下的每个页面单独描述，不糅杂到一起

**输出格式**：

```
【功能名称：XXX功能】

### 1. 列表页面

#### 1.1 页面按钮
- 新增按钮
- 编辑按钮
- 删除按钮
- 导出按钮

#### 1.2 页面查询条件
| 字段名称 | 控件类型 | 说明 |
|----------|----------|------|
| 场所名称 | 文本框 | 模糊查询 |
| 场所分类 | 下拉框 | 选项：企业/个体/政府 |

#### 1.3 页面展示字段
| 字段名称 | 控件类型 | 说明 |
|----------|----------|------|
| 场所名称 | 文本 | - |
| 状态 | 标签 | 草稿/在用 |

#### 1.4 业务规则
- 删除仅草稿状态可操作

#### 1.5 特殊需求
- 导出Excel：是

---

### 2. 新增/编辑弹窗

#### 2.1 页面按钮
- 保存按钮
- 取消按钮

#### 2.2 表单字段
| 字段名称 | 控件类型 | 必填 | 说明 |
|----------|----------|------|------|
| 场所名称 | 文本框 | 是 | 最大50字符 |
| 场所分类 | 下拉框 | 是 | 选项：企业/个体/政府 |
| 咨询电话 | 文本框 | 否 | 11位手机号 |

#### 2.3 业务规则
- 场所名称同一类型下不能重复
```

### 5.2 页面字段分析【重点】

> **说明**：以下字段类型为**参考示例**，不是固定列表。AI应根据实际需求描述灵活识别字段类型。

**常见字段类型分类（参考示例）**：

| 字段类型 | 说明 | 示例 |
|----------|------|------|
| 文本框 | 单行文本输入 | 名称、地址 |
| 下拉框 | 下拉选择 | 分类、状态 |
| 单选按钮组 | 单项选择 | 是/否、性别 |
| 复选框 | 多项选择 | 权限、标签 |
| 日期选择器 | 日期输入 | 创建时间 |
| 上传控件 | 文件/图片上传 | 附件、头像 |
| 文本域 | 多行文本 | 备注、描述 |
| 数字输入 | 数值输入 | 数量、金额 |
| 树选择 | 树形结构选择 | 组织架构、分类 |

**字段类型识别原则**：

1. **根据需求描述识别**：如"展示为下拉框"→下拉框、"单选按钮组"→单选按钮组
2. **根据语义推断**：如"是否"→单选按钮组、"多选"→复选框
3. **未知类型记录为文本**：如无法判断，记录为"文本框"并在不确定项中确认

**灵活处理**：
- 如果遇到未列出的字段类型，按语义最接近的类别归类
- 在不确定项中向用户确认具体控件类型

**分析格式**：

```
【功能：场所信息管理】

列表页：
- 表格列：场所名称、场所类型、负责人、联系电话、状态
- 查询条件：场所名称（模糊查询）、场所类型（下拉框）、状态（下拉框）

新增/编辑弹窗：
- 场所名称：文本框，必填，最大50字符
- 场所类型：下拉框，必填，选项：企业/个体/政府
- 负责人：文本框，必填
- 联系电话：文本框，可选，11位手机号
- 经营状态：单选按钮组，必填，营业中/已停业
- 场所图片：上传控件，可选，支持jpg/png
- 备注：文本域，可选，最大500字符
```

**分析要点**：
1. 区分列表页字段和表单字段
2. 明确字段的输入方式（文本框/下拉框/单选等）
3. 标注必填/可选
4. 标注验证规则（最大长度、格式等）
5. 特殊控件（上传、日期、树选择等）

### 5.3 业务规则分析【重点】

> **说明**：业务规则是需求分析的核心组成部分，用于描述功能的约束条件和逻辑。

**业务规则分类**：

| 类别 | 说明 | 示例 |
|------|------|------|
| 字段规则 | 字段的取值限制 | 最大长度、格式要求、取值范围 |
| 唯一性规则 | 字段是否需要唯一 | 名称在同一类型下唯一 |
| 必填规则 | 字段是否必填 | 必填/可选 |
| 状态规则 | 状态流转规则 | 审核通过后才能生效 |
| 权限规则 | 数据权限控制 | 只能查看/修改自己的数据 |

**业务规则识别要点**：

1. **数值类**：取值范围（大于0、小于100等）
2. **字符串类**：长度限制、格式要求（手机号、邮箱等）
3. **唯一性**：是否需要唯一
4. **必填性**：是否为必填字段
5. **状态流转**：状态变化规则（如：草稿→审核中→已审核）

**分析格式**：

```
【功能：场所信息管理】

业务规则/约束条件：
- 场所名称：最大50字符，同一区域下不能重复
- 联系电话：11位手机号或固定电话（带区号）
- 场所类型：选择后关联不同的资质要求
```

### 5.4 特殊需求分析【重点】

> **说明**：特殊需求包括批量操作、导入导出、消息通知、操作日志等。

**特殊需求分类**：

| 类别 | 说明 | 示例 |
|------|------|------|
| 批量操作 | 批量处理数据 | 批量删除、批量审核、批量导入 |
| 导入导出 | 数据导入导出 | Excel导入、Excel导出 |
| 消息通知 | 站内消息推送 | 站内信、短信、邮件 |
| 操作日志 | 数据变更记录 | 新增、修改、删除记录 |
| 数据关联 | 关联现有模块 | 关联用户表、组织架构表 |

**分析格式**：

```
【功能：场所信息管理】

批量操作：
- 批量删除：是
- 批量审核：是
- 批量导入/导出：是

导入导出：
- 导入：是，Excel模板
- 导出：是，Excel格式

消息通知：
- 站内信：是，触发条件：新增成功通知管理员
- 短信：否
- 邮件：否

操作日志：
- 是否记录：是
- 记录范围：新增、修改、删除

数据关联：
- 关联模块：用户表、组织架构表
- 关联说明：创建时关联当前登录用户
```

### 5.5 优先级定义

| 优先级 | 定义 | 说明 |
|--------|------|------|
| 高 | 必须有 | 核心功能，缺失则需求无法成立 |
| 中 | 应该有 | 重要功能，提升用户体验 |
| 低 | 可以有 | 锦上添花，可后续迭代 |

## 6. 不确定项处理

### 6.1 常见不确定项类型

| 类型 | 示例问题 |
|------|----------|
| 功能边界 | 是否需要支持批量操作？ |
| 数据来源 | 数据来源是新建还是对接现有系统？ |
| 角色权限 | 是否需要区分不同角色？ |
| 操作权限 | 具体操作权限级别？ |
| 流程细节 | 具体业务流程步骤？ |

### 6.2 确认原则

1. **不回避问题**：识别到不确定项必须确认
2. **逐步确认**：每次不超过4个问题
3. **明确选项**：提供明确的选项供用户选择
4. **记录答案**：将确认结果记录到文档

---

## 7. 分析流程

### 7.1 五步流程

```
步骤0: 功能类型确认 → 步骤1: 需求分析 → 步骤2: 不确定项确认 → 步骤3: 结果确认 → 步骤4: 生成文档
```

### 7.2 关键原则

1. **循序渐进**：按步骤顺序执行
2. **门禁机制**：每步必须获得确认
3. **聚焦业务**：只分析业务需求，不涉及技术实现
